# 看懂 Git：版本控制入门

> 面向设计师：理解 Git 不是为了变成工程师，而是为了看懂 AI 和开发同学在「保存、协作、上线」时到底在做什么，配合起来更顺。

作为设计师，你大概率见过这样的文件夹：`首页_终版`、`首页_真的终版`、`首页_最终版_改`。Git 要解决的就是这个问题——只不过它做得更彻底、更可靠。

## 什么是版本控制

版本控制（Version Control）就是**给文件的每一次改动都留一个可回溯的快照**。任何时候你都能：

- 回到过去任意一个版本
- 看清每个版本「谁、什么时候、改了什么」
- 多个人同时改同一个项目而不互相覆盖

对设计师的类比：想象你的设计稿有一条**完整的历史时间线**，每存一次都自动打好标签、写好备注，永远不会丢、永远找得回。这就是 Git 在为代码做的事。

> 别再用文件名管理版本（终版、最终版2）。版本这件事，交给工具。

## Git 的四个区域

Git 把文件分在四个「区域」之间流动。理解了这条流水线，命令就不再是黑魔法：

| 区域 | 是什么 | 设计类比 |
| --- | --- | --- |
| 工作区 | 你正在编辑的文件 | 当前画布 |
| 暂存区 | 准备打包提交的改动 | 选中要导出的图层 |
| 本地仓库 | 你电脑上的版本历史 | 本机的历史版本记录 |
| 远程仓库 | 云端的共享版本库 | 团队协作云盘 |

## 日常基础操作

九成的日子里，你（或 AI / 开发同学）只在重复这一条循环：

1. 在**工作区**改文件
2. `git add` 把改动放进**暂存区**
3. `git commit` 存成一个**本地版本**（带一句说明）
4. `git push` 上传到**远程**，别人才看得到

开工前记得先 `git pull`，把同事的最新改动同步下来，能省掉大量冲突。

## 分支：并行开发不打架

分支（Branch）= **复制一条独立的时间线去做改动，不影响主线**。

类比：你想给首页做个大改版，又不确定行不行——于是**复制一份设计稿单独折腾**，满意了再替换原稿，不满意直接扔掉，原稿毫发无伤。Git 的分支就是这个意思，只是切换、合并都是一条命令的事。

- `main`（主分支）：通常是「稳定、能上线」的那条线
- 功能分支：每个新功能 / 实验单独开一条，做完再合并回 `main`

> 改之前先确认自己在哪条分支上，别直接在 `main` 上乱改。

## 和开发同学协作

当你听到开发说「我提个 PR」「我 merge 一下」「先 rebase 再说」——现在你大概能猜到：他们在把某条分支的改动，安全地并回主线。

你不需要会敲这些命令，但**看懂这套流程**，就能更准确地表达诉求：「这个版本能不能回退？」「这个改动上线了没？」「在哪条分支上改的？」——和工程师的沟通会顺畅很多。
